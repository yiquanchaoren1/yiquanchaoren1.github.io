微处理器

乱序执行和寄存器重命名

这两个技术本质上都是为了解决CPU 流水线停顿的问题，最大化利用硬件单元。

首先是寄存器重命名。它是乱序执行的前提。 汇编里的通用寄存器（如 RAX, RBX）太少（架构寄存器，ARF），导致不同逻辑的指令可能会复用同一个寄存器名（比如都用 EAX）。这会产生 WAW（写后写） 和 WAR（读后写） 这种假依赖。 但 CPU 内部物理寄存器（PRF）其实很多，CPU 通过寄存器重命名技术，将这些架构寄存器映射到内部几百个物理寄存器上。这样一来，两条原本因为名字冲突必须排队的指令，就可以解耦，变成互不相关的操作，为后面的乱序执行扫清了障碍。

接下来是乱序执行。 在解决了名字冲突后，CPU 维护一个保留站 (Reservation Station)。指令不需要按照程序顺序执行，而是遵循Data Flow（数据流）——只要一条指令的操作数准备好了（比如数据从 Cache 回来了），并且运算单元空闲，它就可以立即执行。 这样，当一条指令卡在内存读取（Cache Miss）时，CPU 不会傻等，而是会去执行后面那些和它没依赖关系的指令，从而掩盖了延迟。

现代微处理器架构

对于现代微处理器架构，我理解它本质上是一个为了掩盖内存延迟和挖掘并行度而设计的复杂系统。

可以把它看作一条高度优化的工厂流水线：

  1. 前端（Frontend）是调度员：它负责取指和解码。这里最关键的是分支预测器，它能保证流水线不中断。对于 x86 架构，它还会把指令翻译成微指令（uOps）。
  2. 后端（Backend）是执行工人：这里采用了乱序执行和寄存器重命名技术。通过 ROB（重排序缓冲区）和统一的物理寄存器堆，CPU 能够打破指令间的假依赖，只要数据准备好了就立即计算，填满流水线的空隙。同时，后端配备了宽阔的 SIMD 执行单元（如 AVX2/AVX-512），这是我们做 HPC 向量化加速的基础。
  3. 存储子系统是供应链：因为 CPU 算得太快了，内存太慢，所以设计了 L1/L2/L3 多级缓存。我们在编程时关注的 Cache Line 对齐、False Sharing、以及 TLB Miss，都是为了优化这一部分的效率。

总的来说，现代架构就是在尽量减少 Pipeline Stall（流水线停顿），让计算单元时刻保持满载。”

static

static 修饰全局变量

static 修饰全局变量可以将变量的作用域限定在当前文件中，使得其他文件无法访问该变量。

同时，static 修饰的全局变量在程序启动时被初始化（可以简单理解为在执行 main 函数之前，会执行一个全局的初始化函数，在那里会执行全局变量的初始化），生命周期和程序一样长。

static 修饰局部变量

static 修饰局部变量可以使得变量在函数调用结束后不会被销毁，而是一直存在于内存中，下次调用该函数时可以继续使用。

同时，由于 static 修饰的局部变量的作用域仅限于函数内部，所以其他函数无法访问该变量。

static 修饰函数

static 修饰函数可以将函数的作用域限定在当前文件中，使得其他文件无法访问该函数。

同时，由于 static 修饰的函数只能在当前文件中被调用，因此可以避免命名冲突和代码重复定义。

static 修饰类成员变量和函数

static 修饰类成员变量和函数可以使得它们在所有类对象中共享，且不需要创建对象就可以直接访问。

字节对齐

在C/C++中，字节对齐是内存分配的一种策略。当分配内存时，编译器会自动调整数据结构的内存布局，使得数据成员的起始地址与其自然对齐边界（一般为自己大小的倍数）相匹配

字节对齐的作用和好处

理论上，任何类型的变量都可以从任意地址开始存放。

然而实际上，访问特定类型的变量通常需要从特定对齐的内存地址开始。

因为如果不对数据存储进行适当的对齐，可能会导致存取效率降低。

所以，各种数据类型需要按照一定的规则在内存中排列（起始地址），而不是顺序地一个接一个排放，这种排列就是字节对齐。

例如，有些平台每次读取都是从偶数地址开始。如果一个 int 类型（假设为 32 位系统）存储在偶数地址开始的位置，那么一个读周期就可以读取这 32 位。

但如果存储在奇数地址开始的位置，则需要两个读周期，并将两次读取的结果的高低字节拼凑才能得到这 32 位数据。

显然这会显著降低读取效率。

总结： 字节对齐有助于提高内存访问速度，因为许多处理器都优化了对齐数据的访问。但是，这可能会导致内存中的一些空间浪费。

字节对齐规则

1.自然边界对齐

对于基本数据类型，其自然对齐边界通常为其大小。

例如，char 类型的自然对齐边界为 1 字节，short 为 2 字节，int 和 float 为 4 字节，double 和 64 位指针为 8 字节。具体数值可能因编译器和平台而异。

2.结构体对齐

结构体内部的每个成员都根据其自然对齐边界进行对齐。

也就是可能在成员之间插入填充字节。

结构体本身的总大小也会根据其最大对齐边界的成员进行对齐（比如结构体成员包含的最长类型为int类型，那么整个结构体要按照4的倍数对齐），以便在数组中正确对齐。

3.union对齐

联合体的对齐边界取决于其最大对齐边界的成员。联合体的大小等于其最大大小的成员，因为联合体的所有成员共享相同的内存空间。

4.编译器指令

可以使用编译器指令（如 #pragma pack）更改默认的对齐规则。这个命令是全局生效的（一旦你在代码里写了 #pragma pack(1)，之后定义的所有结构体都会按照这个规则来挤压内存，直到你显式地改回来）。这可以用于减小数据结构的大小，但可能会降低访问性能。

当你使用 #pragma pack(1) 时，你是在告诉编译器：“别管什么读取速度了，把所有的空隙都给我挤掉，哪怕数据跨块了也没关系。”

5.对齐属性

在 C++11 及更高版本中，可以使用 alignas 关键字为数据结构或变量指定对齐要求。这个命令是对某个类型或者对象生效的。例如， 将确保 x 的地址是 16 的倍数。

6.动态内存分配

大多数内存分配函数（如 malloc 和 new）会自动分配足够对齐的内存，以满足任何数据类型的对齐要求。

案例

什么是大端序、小端序和网络字节序，为什么要有字节序？

这是一个非常经典的问题。要理解字节序，我们需要先把内存想象成一排带编号的“抽屉”，每个抽屉（地址）只能存 1 个字节（8位）。

但是，现实中很多数据（如 int, long, float）往往占用 多个字节。当你把一个多字节的数据存入内存时，就产生了一个逻辑问题：你是先把它的“头”放进去，还是先把它的“尾”放进去？

什么是大端和小端？

假设我们有一个 16 进制的整数：0x12345678（占 4 个字节）。

  • 高位字节（头）：12
  • 低位字节（尾）：78
大端序 (Big-Endian)
  • 规则：高位字节存放在低地址，低位字节存放在高地址。
  • 直观理解：符合人类的阅读习惯。从左往右读，地址从小往大。
  • 内存布局：
    地址 0x100 0x101 0x102 0x103
    数据 12 34 56 78
小端序 (Little-Endian)

  • 规则：低位字节存放在低地址，高位字节存放在高地址。

  • 直观理解：“小端在前”（小尾巴放在前面）。这是目前 x86 (Intel/AMD) 架构 CPU 普遍采用的方式。

  • 内存布局：

    地址 0x100 0x101 0x102 0x103
    数据 78 56 34 12

什么是网络字节序？

在网络通讯中，如果一台“大端机器”发了一个 0x12345678 给“小端机器”，而大家如果不统一标准，小端机器接过去解析出来可能就变成了 0x78563412，数据就全乱了。

为了解决兼容性问题：

  • TCP/IP 协议规定：所有网络协议头部（IP头、端口号等）必须使用大端序传递。
  • 因此，网络字节序 = 大端序。

核心概念（定义）

“字节序（Byte Order）是指多字节数据（如 int 或 long）在内存中的存放顺序。

  • 大端序（Big-Endian）：是将数据的高位字节存放在内存的低地址端。这符合人类的阅读习惯，就像我们读数字时，高位在前。
  • 小端序（Little-Endian）：是将数据的低位字节存放在内存的低地址端。目前主流的 x86 和 ARM（默认）架构 CPU 都采用小端序。”

为什么要有字节序？

“之所以存在两种不同的字节序，主要是因为 硬件设计效率 的权衡：

  1. 小端序的计算优势：在计算机进行加法或乘法运算时，通常是从低位算到高位（先算个位）。小端序让 CPU 能够从低地址起始处直接获取低位数据，进位处理起来更自然、效率更高。
  2. 强制类型转换效率：比如将 int 强制转为 short，在小端序下，只需直接截取低地址的数据即可，不需要移动指针。
  3. 大端序的判别优势：大端序读取第一个字节就能立刻判断数字的正负符号。”

inline

当一个函数被声明为 inline 时，编译器会尝试在每一个调用该函数的地方，直接把函数的代码“复制粘贴”进去，而不是像普通函数那样进行传统的“函数调用”。

为什么需要 inline？（原理与优势）

普通函数的开销

在普通的函数调用中，CPU 需要执行一系列复杂操作：

  1. 保存现场：将当前函数的寄存器状态压入栈中。
  2. 跳转：跳到目标函数的内存地址执行代码。
  3. 传参和返回：处理参数传递和返回值。
  4. 恢复现场：执行完后跳回原处并恢复栈状态。

如果函数体本身非常短（例如只有一行代码），那么这些“额外动作”耗费的时间可能比执行函数代码本身还要长。

Inline 的优势

  • 消除开销：省去了压栈、跳转、返回等所有动作。
  • 编译器优化：代码展开后，编译器可以结合上下文进行更好的指令优化。

inline 函数是为了消除函数调用开销而设计的。它建议编译器将函数体在调用处展开。它比宏更安全，因为它有类型检查。但它会导致代码膨胀，所以通常只用于代码极短且频繁调用的函数。最后要记住，内联只是建议，最终决定权在编译器手里

四种强制类型转换

  • static_cast
  • dynamic_cast
  • const_cast
  • reinterpret_cast

1.static_cast（最常用）

  • 用途：用于良性、编译期可确定的类型转换。
  • 场景：
    • 基本数据类型转换（如 int 转 double）。
    • 非多态继承体系中，基类和派生类之间的指针或引用转换（上行转换安全，下行转换不安全）。

2.dynamic_cast（最安全）

  • 用途：用于多态体系下的运行期类型检查。
  • 特性：会检查转换是否有效。如果失败，指针返回 nullptr，引用抛出异常。
  • 要求：基类必须有虚函数（即有虚函数表 vtable）。
  • 场景：安全的向下转型（Downcasting），即把基类指针转为派生类指针。

3. const_cast（唯一能去 const）

  • 用途：专门用于修改类型的 const 或 volatile 属性。
  • 场景：
    • 去掉对象的只读属性，以便调用某些接收非 const 参数的旧接口（注意：修改原本定义为 const 的变量是未定义行为）。

4. reinterpret_cast（最危险）

  • 用途：底层的重新解释。它不改变二进制位，只是换个“眼镜”看内存。
  • 场景：
    • 指针与整数之间的转换。
    • 无关类型指针之间的转换（如 char* 转 int*）。
  • 注意：极易出错，完全由程序员保证安全性。

面向对象

面向对象三大特性

C++ 面向对象编程 (OOP) 的三大特性包括：封装、继承和多态。

封装

封装是将数据（属性）和操作这些数据的函数（方法）组合在一个类（Class）中的过程。

封装的主要目的是隐藏类的内部实现细节，仅暴露必要的接口给外部。

通过封装，我们可以控制类成员的访问级别（例如：public、protected 和 private），限制对类内部数据的直接访问，确保数据的完整性和安全性。

继承

继承是一个类（派生类，Derived Class）从另一个类（基类，Base Class）那里获得其属性和方法的过程。

继承允许我们创建具有共享代码的类层次结构，减少重复代码，提高代码复用性和可维护性。

在 C++ 中，访问修饰符（如 public、protected、private）控制了派生类对基类成员的访问权限。

多态

不同的类对象使用同一方法可能表现为不同的行为

多态是允许不同类的对象使用相同的接口名字，但具有不同实现的特性。

在 C++ 中，多态主要通过虚函数（Virtual Function）和抽象基类（Abstract Base Class）来实现。

虚函数允许在派生类中重写基类的方法，而抽象基类包含至少一个纯虚函数（Pure Virtual Function），不能被实例化，只能作为其他派生类的基类。

通过多态，我们可以编写更加通用、可扩展的代码，提高代码的灵活性。

总结：封装、继承和多态是面向对象编程的三大核心特性，能够帮助我们编写更加模块化、可重用和可维护的代码。

重载、重写和隐藏的区别详解

重载（overloading）：区别一个作用域内方法名相同，但是参数不同的多个函数。

重写（overriding）：子类重写父类的方法。当派生类需要改变或扩展基类方法的功能时，就需要用到重写。

隐藏(hiding)：隐藏是指派生类的函数屏蔽了与其同名的基类函数。注意只要同名函数，不管参数列表是否相同，基类函数都会被隐藏。也就是说重写一定隐藏。

重载和重写的区别

  • 范围区别：重写和被重写的函数在不同的类中，重载和被重载的函数在同一类中（同一作用域）。
  • 参数区别：重写与被重写的函数参数列表一定相同，重载和被重载的函数参数列表一定不同。
  • virtual的区别：重写的基类必须要有virtual修饰，重载函数和被重载函数可以被virtual修饰，也可以没有。

隐藏和重写，重载的区别

  • 与重载范围不同：隐藏函数和被隐藏函数在不同类中。
  • 参数的区别：隐藏函数和被隐藏函数参数列表可以相同，也可以不同，但函数名一定同；当参数不同时，无论基类中的函数是否被virtual修饰，基类函数都是被隐藏，而不是被重写。重写必须参数相同。

C++多态的实现方式详解

C++中的多态是指同一个函数或者操作在不同的对象上有不同的表现形式。

C++实现多态的方法主要包括虚函数、纯虚函数和模板函数

其中虚函数、纯虚函数实现的多态叫动态多态，模板函数、重载等实现的叫静态多态。

区分静态多态和动态多态的一个方法就是看决定所调用的具体方法是在编译期还是运行时，运行时就叫动态多态。

1.虚函数、纯虚函数实现多态

知识点：

virtual 关键字是实现 多态（Polymorphism） 的灵魂。它的核心作用是告诉编译器：这个函数的调用不应该在编译时死锁，而应该在程序运行时根据对象的“真实身份”来决定。

虚析构函数（防止内存泄漏）：

这是面试中的 必考点。当一个类可能被继承时，基类的析构函数必须声明为 virtual。

  • 如果不加 virtual：当你通过基类指针删除一个派生类对象时（如 delete ptr;），编译器只会调用基类的析构函数，而派生类特有的成员（如在构造函数里 new 出来的内存）就不会被释放，导致内存泄漏。
  • 加了 virtual：delete 时会先寻找最底层的派生类析构函数，然后逐级向上调用，确保对象被完整销毁。
虚函数、纯虚函数实现多态：

当我们使用基类指针或引用指向派生类对象时，通过虚函数的机制，可以调用到派生类中重写的函数，从而实现多态。

C++ 的多态必须满足两个条件：

  • 必须通过基类的指针或者引用调用虚函数
  • 被调用的函数是虚函数，且必须完成对基类虚函数的重写

这里解释：为什么必须通过基类的指针或者引用调用虚函数？如果你直接用对象名去调用函数，编译器在编译阶段就死锁了调用哪个函数（静态绑定）；只有当你通过基类的指针或引用去指向一个子类对象时，C++ 的“动态绑定”机制才会启动，在运行时去查找并调用子类重写的版本。

  2. 取出该对象头部的 vptr 指针。
  3. 通过 vptr 找到 Derived 类的 虚函数表。
  4. 从表中索引为 0 的位置取出函数地址。
  5. 发现地址指向的是 Derived::func1，执行它。
4. 关键特性与面试考点
  • 性能开销：虚函数调用比普通函数慢一点点。因为普通函数是“直接跳转”，而虚函数需要“两次寻址”（找 vptr，再找 V-Table）。
  • 对象大小：如果你在一个类里增加了第一个虚函数，你会发现 sizeof(对象) 增大了（在 64 位系统上通常增大 8 字节），这就是因为多了一个 vptr。
  • 构造函数不能是虚的：因为虚函数调用依赖 vptr，而 vptr 是在构造函数执行期间才初始化的。
  • 析构函数应该是虚的：这样才能通过虚函数表找到正确的子类析构函数，防止内存泄漏。

c++对象模型

C++ 对象模型（Object Model）是一个非常深奥且核心的话题，它研究的是：在内存中，C++ 是如何组织类成员（数据和函数）以及如何实现继承、多态等特性的。

理解它，能让你从底层彻底搞清楚 sizeof 的结果、虚函数调用的本质以及继承的开销。

1. 简单对象模型：数据成员的排列

在最基本的情况下（非继承、无虚函数），C++ 对象的内存布局非常简单：它只存储非静态成员变量（Non-static Data Members）。

  • 成员变量：按照在类中声明的顺序排列。
  • 静态成员：存放在全局/静态存储区，不占用具体对象的空间。
  • 成员函数：存放在代码段（Code Segment），所有对象共享同一份函数代码，也不占用对象空间。
  • 内存对齐：为了提高 CPU 读取效率，编译器会在成员之间填充一些字节（Padding），使得对象大小通常是 4 或 8 的倍数。

2. 核心：带虚函数的对象模型

当你为一个类添加了 virtual 关键字，对象模型就会发生质变。为了支持多态，编译器引入了两个核心组件：

  1. 虚函数表（vtable）：每个包含虚函数的类都有一张表，存放该类所有虚函数的地址。
  2. 虚表指针（vptr）：每个对象实体的头部（或尾部，取决于编译器）会多出一个指针，指向该类的 vtable。

此时对象的物理大小： 成员变量总和 + vptr的大小（64位系统为8字节） + 内存对齐。

3. 继承下的对象模型

(1) 单继承

子类的对象模型包含两部分：父类的所有数据成员 + 子类自己的数据成员。

  • 如果父类有虚函数，子类会继承父类的 vptr。
  • 如果子类重写（Override）了虚函数，子类 vtable 中对应的地址会被替换为子类函数的地址。
(2) 多重继承

如果一个类继承自多个父类（如 class C : public A, public B），对象内部会包含多个 vptr。

  • 每个基类子对象都有自己的 vptr，分别指向对应的虚函数表。
  • 这保证了当你把 C* 强转为 A* 或 B* 时，都能正确找到对应的虚函数。
(3) 虚继承（解决菱形继承）

虚继承的对象模型最复杂。它引入了 虚基类表指针（vbptr）。

  • 虚基类的成员被放在对象内存的最末端。
  • 子类通过 vbptr 找到虚基类成员的偏移量。
  • 目的：确保无论经过多少次继承，虚基类（如 Animal）在最底层的子类（如 Liger）中只有一份拷贝。

4. 总结：一个 C++ 对象里到底有什么？

我们可以把一个复杂的 C++ 对象想象成一叠书：

  • 底层：所有父类的数据成员（按继承顺序）。
  • 夹层：一个或多个 vptr（如果有虚函数）。
  • 顶层：自己定义的数据成员。
  • 隐藏组件：为了对齐而存在的空隙（Padding）。

5. 面试中的常见问题：

  • Q: 为什么空类的大小是 1？
    • A: 为了确保每个对象在内存中都有唯一的地址。如果大小为 0，连续创建两个对象将无法区分它们的地址。
  • Q: 成员函数的地址存在对象里吗？
    • A: 不在。普通成员函数在代码段；虚函数地址在类关联的 vtable 里。对象里只存一个指向 vtable 的指针（vptr）。
  • Q: 构造函数可以是虚函数吗？
    • A: 不行。因为 vptr 是在构造函数执行期间才初始化的。在对象还没构造好之前，根本没有 vptr 去查虚函数表。

纯虚函数

纯虚函数是一种在基类中声明但没有实现的虚函数。

它的作用是定义了一种接口，这个接口需要由派生类来实现。类似于java中的接口

包含纯虚函数的类称为抽象类（Abstract Class）。

抽象类仅仅提供了一些接口，但是没有实现具体的功能。

作用就是制定各种接口，通过派生类来实现不同的功能，从而实现代码的复用和可扩展性。

另外，抽象类无法实例化，也就是无法创建对象。

原因很简单，纯虚函数没有函数体，不是完整的函数，无法调用，也无法为其分配内存空间。

为什么 C++ 构造函数不能是虚函数？

1. 从语法层面来说

虚函数的主要目的是实现多态，即允许在派生类中覆盖基类的成员函数。

但是，构造函数负责初始化类的对象，每个类都应该有自己的构造函数。

在派生类中，基类的构造函数会被自动调用，用于初始化基类的成员。因此，构造函数没有被覆盖的必要，不需要使用虚函数来实现多态。

2. 从虚函数表机制回答

虚函数使用了一种称为虚函数表（vtable）的机制。然而，在调用构造函数时，对象还没有完全创建和初始化，所以虚函数表可能尚未设置。

这意味着在构造函数中使用虚函数表会导致未定义的行为。

只有执行完了对象的构造，虚函数表才会被正确的初始化。

为什么 C++ 基类析构函数需要是虚函数？

C++中，基类的析构函数需要定义为虚函数，以确保在通过基类指针或引用删除派生类对象时，能够正确地调用派生类的析构函数，否则派生类的析构函数不会被调用，这部分资源也就并无法被释放。

将基类的析构函数定义为虚函数后，C++运行时会自动识别指向的对象的实际类型，并确保调用正确的派生类析构函数。当派生类析构函数执行完毕后，基类的析构函数也会自动调用，以确保对象完全销毁。

为什么默认的析构函数不是虚函数

那么既然基类的析构函数如此有必要被定义成虚函数，为何类的默认析构函数却是非虚函数呢？

首先一点，语言设计者如此设计，肯定是有道理的。

原来是因为，虚函数不同于普通成员函数，当类中有虚成员函数时，类会自动进行一些额外工作。

这些额外的工作包括生成虚函数表和虚表指针，虚表指针指向虚函数表。

每个类都有自己的虚函数表，虚函数表的作用就是保存本类中虚函数的地址，我们可以把虚函数表形象地看成一个数组，这个数组的每个元素存放的就是各个虚函数的地址。

这样一来，就会占用额外的内存，当们定义的类不被其他类继承时，这种内存开销无疑是浪费的。

这样一说，问题就不言而喻了。当我们创建一个类时，系统默认我们不会将该类作为基类，所以就将默认的析构函数定义成非虚函数，这样就不会占用额外的内存空间。

同时，系统也相信程序开发者在定义一个基类时，会显示地将基类的析构函数定义成虚函数，此时该类才会维护虚函数表和虚表指针。

零成本抽象原则

这也就是 C++ 的一个设计哲学：zero overhead abstraction

我的理解，所谓「零成本抽象」有两个层面的意思。

  1. 不需要为没有使用到的语言特性付出代价。
  2. 使用某种语言特性，不会带来运行时的代价。

放在这个地方就是，如果我们知道一个类不会被其它类继承，那么也就没必要将析构函数设置为虚函数，因为一旦引入虚函数就会引入虚表机制，这会造成额外的开销

为什么C++的成员模板函数不能是virtual的？

“C++ 不支持成员模板虚函数，主要是因为虚函数表的长度必须在编译期确定。

虚函数依赖 VTable 的固定偏移量来定位函数地址，而模板函数只有在实例化时才会产生真正的代码。由于编译器在处理类定义时，无法预知该模板会被实例化出多少个版本，因此无法确定 VTable 的大小和索引分配。如果强行支持，将需要链接器进行复杂的全局处理，这不仅违背了 C++ 的编译模型，也会带来巨大的运行时开销。”

因为C++的编译与链接模型是”分离”的(至少是部分原因吧)。

  • 从Unix/C开始，一个C/C++程序就可以被分开编译，然后用一个linker链接起来。这种模型有一个问题，就是各个编译单元可能对另一个编译单元一无所知。
  • 一个 function template最后到底会被 instantiate 为多少个函数，要等整个程序(所有的编译单元)全部被编译完成才知道。
  • 同时，virtual function的实现大多利用了一个”虚函数表”的东西，这种实现中，一个类的内存布局(或者说虚函数表的内存布局)需要在这个类编译完成的时候就被完全确定。

所以当一个虚函数是模板函数时，编译器在编译时无法为其生成一个确定的虚函数表条目，因为模板函数可以有无数个实例。但是编译时无法确定需要调用哪个特定的模板实例。因此，C++标准规定member function 不能既是 template 又是 virtual 的。

模板（Templates）是如何工作的？

C++ 模板的本质是“代码生成器”。它遵循的是 “按需实例化” 的原则。

  • 编译前的蓝图：在你调用模板之前，它只是一段语义检查过的“模板代码”，并不占用内存，也不会生成二进制指令。
  • 实例化（Instantiation）：当你写下 vector<int> 或调用 add<float>(a, b) 时，编译器会根据你提供的参数（int 或 float），将模板中的占位符替换掉，产生一份全新的、具体的函数或类代码。
  • 二次编译：编译器会对生成的具体代码进行第二次编译，产生目标文件。
  • 代码膨胀风险：由于每个不同的参数类型都会生成一份独立的代码（比如 Stack<int> 和 Stack<double> 是两个完全不同的类），过度使用模板可能导致生成的二进制文件体积显著增大。

从源代码到可执行文件的过程

这是一个四阶段的流水线：预处理 -> 编译 -> 汇编 -> 链接。

第一阶段：预处理 (Preprocessing)

  • 输入：.cpp 和 .h 文件。
  • 操作：处理 #include（把头文件内容复制进来）、处理宏定义（#define 文本替换）、处理条件编译（#ifdef）。
  • 输出：.i 文本文件。

第二阶段：编译 (Compilation)

  • 操作：进行语法、词法分析，优化代码。最重要的任务是将高级语言转化为汇编语言。
  • 输出：.s 汇编文件。

第三阶段：汇编 (Assembly)

  • 操作：将汇编指令逐条翻译成机器指令（二进制）。
  • 输出：.obj (Windows) 或 .o (Linux) 目标文件。此时文件已经是二进制，但还不能运行，因为它内部引用的外部函数地址（如 printf）还是空的。

第四阶段：链接 (Linking)

  • 操作：
    • 地址重定向：把各个目标文件拼接在一起。
    • 符号解析：把代码中引用的符号（变量名、函数名）关联到正确的内存地址。
    • 库合并：把标准库或第三方库（.lib / .a）打包进来。
  • 输出：.exe 或可执行二进制文件。

内存管理

内存布局

程序运行时，代码、数据等都存放在不同的内存区域，这些内存区域从逻辑上做了划分，大概以下几个区域：代码区、全局/静态存储区、栈区、堆区和常量区。

1.代码区：也就是 .text 段， 代码区存放程序的二进制代码，它是只读的，以防止程序在运行过程中被意外修改。

2.全局/静态存储区：全局变量和静态变量都存放在全局/静态存储区。

以前在 C 语言中全局变量又分为初始化的和未初始化的，分别放在上面图中的 .bss 和 .data 段，但在 C++里面没有这个区分了，他们共同占用同一块内存区，就叫做全局存储区。

这个区域的内存在程序的生命周期几乎都是全局的

3.栈区：栈区用于存储函数调用时的局部变量、函数参数以及返回地址。

4.堆：堆区是用于动态内存分配的区域，当使用new（C++）或者malloc（C）分配内存时，分配的内存块就位于堆区。我们需要手动释放这些内存，否则可能导致内存泄漏。

在 C++ 中，动态分配内存（Dynamic Memory Allocation） 指的是程序在运行时（Runtime）根据实际需要，手动向操作系统申请内存空间的行为。

与之相对的是“静态分配”或“自动分配”（如局部变量），后者在编译时就确定了大小，并在函数结束时自动销毁。

5.常量区

常量区用于存储常量数据，例如字符串字面量和其他编译时常量。这个区域通常也是只读的。

面向过程和面向对象是两种解决问题的思维模型。面向过程是以过程为中心，把问题分解成一系列步骤（函数）；而面向对象是以对象为中心，将数据和行为封装在一起，通过对象间的协作来解决问题

对比

1. 数据的安全性与封装性
  • POP：数据通常是全局的或通过参数传递，任何函数都能修改它，容易出现“牵一发而动全身”的 Bug。
  • OOP：数据被封装在类内部（Private），只能通过特定的接口（Public Method）修改，符合“高内聚、低耦合”。
2. 维护性与扩展性（核心点）
  • POP：随着逻辑变复杂，函数间的依赖会形成“面条代码”，极难维护。
  • OOP：利用继承、封装、多态，可以很方便地增加新功能而不需要改动旧代码（开闭原则）。
3. 执行效率
  • POP：效率更高。因为它直接调用函数，没有对象创建的内存开销和虚函数表查找的运行时损耗。
  • OOP：相对稍慢。需要进行对象实例化、指针跳转等，但对于现代硬件来说，这部分开销通常可以忽略不计。

数组

首先从 p 处开始，先与[]结合，说明 p 是一个数组。然后与 int 结合，说明数组里的元素是整型的，所以 p 是一个由整型数据组成的数组。

即：p is arry(size 3) of int.

指针数组

首先从 p 处开始，先与 [] 结合，因为其优先级比高（ [] 在c语言中属于后缀运算符和 () 等同为最高优先级），所以 p 是一个数组。然后再与 * 结合，说明数组里的元素是指针类型。之后再与int结合，说明指针所指向的内容的类型是整型的，所以 p 是一个指向 int 的指针数组。

英文即: p is arry(size 3) of pointer to int.

数组指针

首先从 p 处开始，先与 * 结合(因为 * 是被括号包围的)，说明 p 是一个指针。然后再与[]结合(与”()”这步可以忽略，只是为了改变优先级)，说明指针所指向的内容是一个数组。之后再与 int 结合，说明数组里的元素是整型的，大小是 3，所以 p 是一个指向 int 数组(大小为3)的指针。

英文即：p is pointer to arry(size 3) of int

函数声明

从 p 处起，先与 () 结合，说明 p 是一个函数。然后进入 () 里分析，说明该函数有一个整型变量的参数，之后再与外面的 int 结合，说明函数的返回值是一个整型数据。

英文即: p is function(int) returning int.

函数指针

从 p 处开始，先与指针结合，说明 p 是一个指针。然后与()结合，说明指针指向的是一个函数。之后再与()里的int结合，说明函数有一个int型的参数，再与最外层的int结合，说明函数的返回类型是整型，所以 p 是一个指向有一个整型参数且返回类型为整型的函数的指针。

英文即: p is pointer to function(int) returning int

指针和引用的区别

指针和引用在 C++ 中都用于间接访问变量，但它们有一些区别：

  1. 指针是一个变量，它保存了另一个变量的内存地址；引用是另一个变量的别名，与原变量共享内存地址。
  2. 指针(除指针常量)可以被重新赋值，指向不同的变量；引用在初始化后不能更改，始终指向同一个变量。
  3. 指针可以为 nullptr，表示不指向任何变量；引用必须绑定到一个变量。
  4. 使用指针需要对其进行解引用以获取或修改其指向的变量的值；引用可以直接使用，无需解引用。

RALL

资源获取即初始化（Resource Acquisition Is Initialization，简称 RAII）是一种 C++ 编程技术，它将在使用前获取（分配的堆内存、执行线程、打开的套接字、打开的文件、锁定的互斥量、磁盘空间、数据库连接等有限资源）的资源的生命周期与某个对象的生命周期绑定在一起。

确保在控制对象的生命周期结束时，按照资源获取的相反顺序释放所有资源。

同样，如果资源获取失败（构造函数退出并带有异常），则按照初始化的相反顺序释放所有已完全构造的成员和基类子对象所获取的资源。

这利用了核心语言特性（对象生命周期、作用域退出、初始化顺序和堆栈展开），以消除资源泄漏并确保异常安全。

RAII 的原理

RAII的核心思想就是：

利用栈上局部变量的自动析构来保证资源一定会被释放。

因为我们平常 C++ 编程过程中，经常会忘了释放资源，比如申请的堆内存忘了手动释放，那么就会导致内存泄露。

还有一些常见是程序遇到了异常，提前终止了，我们的资源也来不及释放。

但是变量的析构函数的调用是由编译器保证的一定会被执行，所以如果资源的获取和释放与对象的构造和析构绑定在一起，就不会有各种资源泄露问题

RALL的实现步骤

一般设计实现一个 RAII 类需要四个步骤:

  • 设计一个类封装资源，资源可以是内存、文件、socket、锁等等一切
  • 在构造函数中执行资源的初始化，比如申请内存、打开文件、申请锁
  • 在析构函数中执行销毁操作，比如释放内存、关闭文件、释放锁
  • 使用时声明一个该对象的类，一般在你希望的作用域声明即可，比如在函数开始，或者作为类的成员变量

在 C++ 中，局部变量（栈对象）有一个特性：一旦离开它的作用域（即遇到最近的一个 }），它就会被自动销毁，从而触发析构函数。

2. 类成员变量：控制“对象存续期间”的资源

这也就是 RAII 的精髓：把手动管理资源（容易出错），转换成管理对象的生命周期（系统自动保证）。

malloc-free 内存分配原理

malloc 和 free 是 C 语言中用于动态内存分配和释放内存的两个函数。

malloc 不是直接系统调用，而是一个用户级内存管理器。它通过预先向操作系统申请大块内存并自行管理，减少了频繁进出内核态的开销（Context Switch）。

底层系统调用：

brk/sbrk

在进程的虚拟内存布局中，堆是从低地址向高地址增长的。

  • start_brk：指向堆的起始位置（固定不变）。
  • **brk (current break)**：指向当前堆的最高边界。

小块内存（通常 < 128KB）—— brk()

  • 原理：通过移动堆顶指针 brk 来分配内存。
  • 特点：分配速度快，但释放后内存不一定会还给操作系统，而是由 malloc 留存以便复用。这可能导致内存碎片。
brk 与 sbrk 的区别

在底层的系统接口中，通常有两个相关的词：

  • **brk(addr)**：直接将“水位线”设置到指定的地址 addr。
  • sbrk(increment)：这是程序员更常用的接口，它在当前水位线的基础上增加 increment 字节。
    • sbrk(0)：这是一个常用的技巧，用来返回当前堆顶的绝对地址。
mmap

1.mmap函数第一种用法是映射磁盘文件到内存中；（把磁盘当内存用）

（1）内存映射段(mmap) 的作用是：内核将硬盘文件的内容直接映射到内存，任何应用程序都可通过 Linux 的 mmap() 系统调用请求这种映射。

（2）内存映射是一种方便高效的文件 I/O 方式， 因而被用于装载动态共享库。

**传统方式 (read/write)**：数据要经过“磁盘 > 内核页缓存 > 用户缓冲区”的两次拷贝，还要进行多次系统调用。

mmap 方式：内核直接把磁盘文件的某一段“投影”到你的进程地址空间里。

物理意义：你修改这块内存，操作系统会在后台自动把改动写回磁盘。

2.而malloc使用的mmap函数的第二种用法，即匿名映射，匿名映射不映射磁盘文件，而是向映射区申请一块内存。 匿名映射：malloc 分配大内存的秘密

当你调用 malloc 申请内存时，底层其实有两个“老板”：

  1. **小额贷款 (brk)**：如果你只申请几十 KB，系统会移动堆（Heap）的边界线。
  2. **大额贷款 (mmap 匿名映射)**：如果你申请的内存超过了阈值（通常是 128KB），malloc 就会调用 mmap。
    • “匿名”是什么意思？ 意思是这块内存不对应任何磁盘文件，它纯粹就是一块物理 RAM。
    • 为什么用 mmap？ 因为 mmap 分配的内存可以独立释放（munmap）。如果一直用 brk 往上推，只要堆顶部的内存没释放，下面的内存即便不用了也还给不了操作系统，会导致内存碎片。

大块内存（通常 > 128KB）—— mmap()

  • 原理：在进程的虚拟地址空间中通过“匿名映射”申请一块独立的内存页。
  • 特点：释放时（free）会直接解除映射，将内存归还给操作系统。

mmap 映射区向下扩展，堆向上扩展，两者相对扩展，直到耗尽虚拟地址空间中的剩余区域。

  • 堆（Heap）：从低地址向高地址长（向上）。
  • 栈（Stack）：从高地址向低地址长（向下）。
  • mmap 映射区：位于堆和栈之间的一块巨大区域。当你在程序里加载 .so 动态库，或者通过 malloc 申请大内存时，它就会在这个区域寻找空位。

这种设计是为了最大化利用虚拟地址空间。堆和 mmap 区中间隔得很远，它们可以各自生长，直到撞到一起。

malloc原理

malloc 的本质是一个用户态内存管理器。它并非每次申请都找内核要内存，而是预先通过系统调用（brk/mmap）申请一大块内存作为内存池，然后在用户态进行二次分配和管理。这样做是为了减少系统调用的开销，提高分配效率

整体用法:

简单对比

  1. 语法不同：malloc/free是一个C语言的函数，而new/delete是C++的运算符。
  2. 分配内存的方式不同：malloc只分配内存，而new会分配内存并且调用对象的构造函数来初始化对象。
  3. 返回值不同：malloc返回一个 void 指针，需要自己强制类型转换，而new返回一个指向对象类型的指针。
  4. malloc 需要传入需要分配的大小，而 new 编译器会自动计算所构造对象的大小

1.new 操作符从自由存储区（free store）上为对象动态分配内存空间，而malloc函数从堆上动态分配内存。

自由存储区是 C++ 基于new操作符的一个抽象概念，凡是通过 new 操作符进行内存申请，该内存即为自由存储区。

而堆是操作系统中的术语，是操作系统所维护的一块特殊内存，用于程序的内存动态分配，C语言使用malloc从堆上分配内存，使用free释放已分配的对应内存。

那么自由存储区是否能够是堆（问题等价于new是否能在堆上动态分配内存），这取决于operator new 的实现细节。

自由存储区不仅可以是堆，还可以是静态存储区，这都看operator new在哪里为对象分配内存。

不过大家可以基本将自由存储区等价于堆区。

2.内存分配失败时返回值

new内存分配失败时，会抛出bac_alloc异常，它不会返回NULL；malloc分配内存失败时返回NULL。

3.是否调用构造函数/析构函数

  • 第一步：调用operator new 函数（对于数组是operator new[]）分配一块足够大的，原始的，未命名的内存空间以便存储特定类型的对象。
  • 第二步：编译器运行相应的构造函数以构造对象，并为其传入初值。
  • 第三部：对象构造完成后，返回一个指向该对象的指针。

使用 malloc 分配的内存后，如果在使用过程中发现内存不足，可以使用 realloc 函数进行内存重新分配实现内存的扩充。

内存泄露如何定位、检测以及避免

内存泄露是什么？

简单来说就是：在程序中申请了动态内存，却没有释放，如果程序长期运行下去，最终会导致没有内存可供分配。

所以不少大厂的服务有个特点，就是会定期重启服务进程，重启的目的就是让操作系统回收整个进程的资源包括内存，这样一点点的内存泄露问题即使无法定位，也不是什么大问题哈哈哈

如何监测？

检测内存泄露的方法：

  1. 手动检查代码：仔细检查代码中的内存分配和释放，确保每次分配内存后都有相应的释放操作。比如 malloc和free、new和delete是否配对使用了。
  2. 使用调试器和工具：有一些工具可以帮助检测内存泄露。例如：
    • Valgrind（仅限于Linux和macOS）：Valgrind是一个功能强大的内存管理分析工具，可以检测内存泄露、未初始化的内存访问、数组越界等问题。使用Valgrind分析程序时，只需在命令行中输入valgrind --leak-check=yes your_program即可。
    • Visual Studio中的CRT（C Runtime）调试功能：Visual Studio提供了一些用于检测内存泄露的C Runtime库调试功能。例如，_CrtDumpMemoryLeaks函数可以在程序结束时报告内存泄露。
    • AddressSanitizer：AddressSanitizer是一个用于检测内存错误的编译器插件，适用于GCC和Clang。要启用AddressSanitizer，只需在编译时添加-fsanitize=address选项。
    • headtrack

如何避免内存泄露

  1. 使用智能指针（C++）：在C++中，可以使用智能指针（如std::unique_ptr和std::shared_ptr）来自动管理内存。这些智能指针在作用域结束时会自动释放所指向的内存，从而降低忘记释放内存或者程序异常导致内存泄露的风险。
  2. 异常安全：在C++中，如果程序抛出异常，需要确保在异常处理过程中正确释放已分配的内存。使用try-catch块来捕获异常并在适当的位置释放内存。或者使用RAII技术

内存错误

日常工作中，往往出现和内存有关的错误，包括各种 segmentfault、coredump等等

1. 内存泄漏 (Memory Leak)
  • 现象：程序运行时间越长，占用的内存就越多，最终导致系统资源耗尽或 OOM（Out of Memory）崩溃。
  • 原因：使用 new 或 malloc 申请了堆内存，但在程序结束或不再使用时，没有对应的 delete 或 free。
  • 典型场景：在循环中不断申请内存，或者在函数提前 return 时跳过了释放逻辑。
2. 野指针与悬空指针 (Wild & Dangling Pointers)
  • **野指针 (Wild Pointer)**：指针在声明时没有初始化，它指向的是一个随机的内存地址。
  • **悬空指针 (Dangling Pointer)**：指针指向的内存已经被 free 或 delete 了，但指针变量本身没有置为 nullptr。
  • 后果：解引用这些指针会导致不可预知的行为，可能导致崩溃，也可能修改了其他变量的数据。
3. 越界访问 (Out-of-Bounds Access)
  • **缓冲区溢出 (Buffer Overflow)**：写入的数据超过了分配的内存空间。比如定义了 int a[10]，却尝试写入 a[10] 或 a[11]。
  • 底层后果：这通常会覆盖栈上的返回地址或堆上的元数据（Header），是黑客实施攻击（如执行恶意 shellcode）的常见入口。
4. 段错误 (Segmentation Fault)

这是 Linux 下最常见的崩溃提示，本质上是进程触发了操作系统的内存保护机制。

  • 常见诱因：
    • 解引用 NULL 指针。
    • 尝试修改只读内存（例如修改字符串常量 char *s = "hello"; s[0] = 'H';）。
    • 访问了不属于当前进程的地址空间（跨过了 brk 围墙）。
5. 重复释放 (Double Free)
  • 现象：对同一块内存地址连续调用两次 free 或 delete。
  • 后果：这会破坏 malloc 管理器内部的空闲链表（Bins）结构，导致程序立即崩溃，或者在随后的内存分配中产生诡异的逻辑错误。
6. 使用未初始化的内存
  • 现象：在堆上分配内存后直接读取，而没有赋值。
  • 原因：malloc 分配的内存块里残留的是上一个使用者留下的“脏数据”。直接读取这些数据会导致程序逻辑出现随机波动。
7. 内存对齐错误 (Alignment Error)
  • 现象：在某些严格要求对齐的硬件架构上（如某些 ARM），如果尝试通过指针读取一个没有对齐到 4 或 8 字节边界的数据，会触发硬件异常。

1. nullptr

为什么现代 C++ 推荐用 nullptr 而不是 NULL？”

“核心意义在于类型安全和消除重载歧义。

NULL 在 C++ 中通常被定义为整数 0，这在函数重载时会导致 int 和 pointer 类型的匹配混乱。而 nullptr 有专门的类型 std::nullptr_t，它只能隐式转换为指针，不能转换为整数。

在开发高性能数值库时，使用 nullptr 能让编译器在编译期就拦截非法赋值，同时提高模板代码的健壮性。这也是我编写求解器时坚持的现代 C++ 规范。”

2.auto

auto 的核心意义是类型推导（Type Inference）。它让编译器在编译期根据变量的初始值自动确定其类型。

  • 减少冗余： 避免书写又长又臭的类型名
  • 重构友好： 如果改变了函数的返回值类型，所有接收该返回值的 auto 变量会自动更新，不需要手动修改。
  • 强制初始化： 使用 auto 声明变量时必须初始化，这在无形中减少了“未初始化变量”带来的 Bug。

5.左值引用、右值引用

C++ 中所有的值都必然属于左值、右值二者之一。左值是指表达式结束后依然存在的持久化对象，右值是指表达式结束时就不再存在的临时对象。所有的具名变量或者对象都是左值，而右值不具名。很难得到左值和右值的真正定义，但是有一个可以区分左值和右值的便捷方法：看能不能对表达式取地址，如果能，则为左值，否则为右值。
