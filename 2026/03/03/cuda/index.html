SM

在一个 SM 中，多组 warp 看起来像是“同时执行”的，但实际上它们是通过硬件的多路快速切换（SIMT warp scheduler）实现的伪并行。一个 warp 的指令在某一周期被执行，下一周期可能换另一个 warp 执行。

例如，在Nvidia H100上，每个SM最多可以处理
32个线程块、
64个warp（即2048个线程）和
每个块1024个线程。

这些都是上限，不能超过上限。比如

eg1.如果我们在申请资源时使用32作为线程块的大小，并总共需要申请使用2048个线程，那么需要64个这样的线程块。然而，每个SM一次只能处理32个块，即使一个SM完全可以运行2048个线程，但由于线程块的限制，每个SM只能运行32*32个线程，因此它需要两个SM来运行，且每个只能占用50%的GPU资源。

eg2.同样，每个SM有65536个寄存器。要同时执行2048个线程，每个线程最多可以有32个寄存器（65536/2048 = 32）。如果在我们程序中一个内核中每个线程需要64个寄存器，那么每个SM上就只能运行1024个线程，同样会导致50%的占用率。

SM的组成

一个SM中可以并行多个线程。通过线程束调度器进行软件层面控制，通过分配单元将线程分配到具体的计算单元中。计算单元分为int、FP32、FP64以及Tensor Core。
对应的功能结构如下：

cuda core

warp 是 32 线程组成的执行单元
这些线程执行 同一条指令
这条指令被送入 CUDA cores
多个 CUDA cores 并行执行这条指令的运算部分
SM 每周期调度几个 warp 取决于调度器数量，不是 core 数

也就是说多个cuda core同时执行一个warp。

warp调度器

Warp 调度器负责在 SM 内决定“下一周期由哪个 warp 执行指令”。
它的作用类似 CPU 中的“线程调度器”，但快得多 —— 每个时钟周期都可能切换一次。

主要功能：

1.选择一个“就绪”的 warp 执行指令

SM 中可能驻留几十个 warp（例如 H100 上最多 64 个 Warp）。
但：

  • 有的 warp 在等待内存（可能需要几百周期）
  • 有的 warp 在等待 barrier
  • 有的 warp 等前一条指令完成

调度器负责从 所有“就绪状态”的 warp 中挑一个来执行。

2.隐藏内存/指令延迟

典型例子：

warp A 访问全局内存 → 需要等待几百个周期
warp B 准备好了 → 调度器选择 warp B 执行
warp C 准备好了 → 下一周期执行 warp C

这样 GPU 永远不会闲着。

这是 GPU 高吞吐的秘密。

GPU 的上下文切换是硬件级别的，无任何开销，比 CPU 跨线程切换快 10^5 倍。

3.分配执行单元（INT/FP/Tensor/Load-Store等）

调度器要分析 warp 的指令类型：

  • FP32 指令 → FP32 ALU
  • INT 指令 → INT ALU
  • 访存指令 → Load/Store 单元
  • Tensor Core 指令 → Tensor Core

如果不同 warp 的指令使用不同执行单元，那么它们可以 同一周期内并行运行。

例如（一个周期内多个 warp 并行）：

  • warp #12 → FP32 单元
  • warp #8 → INT 单元
  • warp #3 → Load/Store 单元
  • warp #5 → Tensor Core

MPS

1MPS相关背景

1.1CUDA计算流程

CUDA 是 NVIDIA 推出的通用并行计算平台与编程模型。它利用 GPU 中大量的并行计算核心，能够比 CPU 更高效地处理复杂的计算任务。一个 CUDA 程序执行时，首先会通过CUDA API或者运行时API来创建一个CUDA上下文，这个上下文包含了GPU内存、Kernel堆栈、执行状态等所有的硬件资源和软件状态。

所有CUDA任务都必须在CUDA流中执行，流是一系列命令（如内核启动、内存拷贝）的顺序队列，不同流中的任务可以并发执行，从而实现粗粒度并行。

在硬件层面，每个CUDA流会映射到GPU上的一个或多个硬件工作队列。支持Hyper-Q技术的GPU拥有多个硬件队列，允许来自同一CUDA上下文的多个流真正并行执行，提升了 GPU 资源利用率。

然而，来自不同 CUDA 上下文的任务只能通过GPU的时间切片调度器轮流执行，无法真正并发。当一个上下文的任务无法占满 GPU 时，就会导致计算资源闲置。这样就引出了MPS。

此外，CUDA 驱动会在后台创建内部线程，用于处理来自操作系统的异步通知，并执行用户定义的回调函数，确保 CPU 与 GPU 之间的协作能高效、异步地进行。

1.2 GPU计算模式（Compute Modes）

可通过nvidia-smi命令设置三种GPU计算模式：

  • DEFAULT：多个进程可以同时使用GPU，每个进程的单独线程可以同时向GPU提交任务。
  • EXCLUSIVE_PROCESS：GPU一次只分配给一个进程，单个进程线程可以并发地向GPU提交工作。
  • PROHIBITED：计算类型的应用无法使用GPU。

当使用MPS时，需要将计算模式修改为EXCLUSIVE_PROCESS，这样保证同一时间只有一个MPS server正在使用该GPU，多个client端连接到这一个server端来使用该GPU，提高了可靠性。

1.3 CUDA context

类似于CPU进程上下文，表示进程在GPU上执行操作所需要和关联的软件状态、资源和环境。每个CUDA上下文包含：

  • 从CPU上统一分配的GPU上的内存空间，即Global Memory（cudaMalloc/cudaMallocManaged）
  • kernel函数中定义和分配的堆栈空间，即GPU Local memory
  • CUDA streams和events对象等
  • 文件句柄和代码

不同进程有自己的CUDA上下文，每个CUDA上下文有自己的地址空间，相互直接不能直接访问。

1.4 GPU Hyper-Q特性

Hyper-Q（Hyper-Queue）是NVIDIA在 Kepler架构（如 Tesla K80）中引入的GPU硬件特性，主要用于解决早期GPU单硬件任务队列的瓶颈问题。在Hyper-Q之前，CPU和GPU之间只有一个硬件任务队列，多个CPU线程或MPI进程向同一个GPU提交任务时会相互阻塞。

Hyper-Q提供32个独立的硬件工作队列，让多个CPU线程或者进程同时向一个GPU提交任务，实现CUDA kernels的并行执行，从而大幅提升GPU利用率和并发性能。可通过环境变量CUDA_DEVICE_MAX_CONNECTIONS=32（默认8）来设置。

Hyper-Q的核心价值：

  • 减少CPU的闲置：多个CPU核心可同时提交任务至GPU。
  • 提升GPU利用率：GPU同时运行多个进程，提高利用率。
  • MPS的基础：Hyper-Q为多进程共享GPU提供了硬件支持

2 Multi-Process Service详细介绍

简单来说就是把多个进程当成一个进程来处理，多个进程退化为线程共享这个大进程的资源，避免cuda 上下文切换。同时利用Hyper-Q实现线程的并行。

2.1 什么是MPS

MPS即Multi-Process Service是一组可替换的、二进制兼容的CUDA API实现。MPS利用GPU上的Hyper-Q功能将多进程CUDA程序协调以透明并行运行。

Hyper-Q是GPU的一个硬件特性，允许多个CPU线程或者进程同时加载任务到一个GPU上，实现CUDA kernel在同一个GPU上并行处理。当单个应用程序进程未充分利用GPU的计算能力时，MPS将能有效提高GPU利用率。

MPS有以下几个组件：

  • Control Daemon Process：控制守护进程负责启动和停止server，以及协调client和server间的连接。
  • Client Runtime：MPS客户端运行时内置于CUDA驱动程序库中，可以由任何CUDA应用程序透明地使用。
  • Server Process：Server端是客户端与GPU的共享连接，提供客户端之间的并行性。

2.2 MPS的三大核心优势

1）提高GPU利用率

如果没有MPS的场景下，运行在GPU上的单个进程并不能充分利用这个GPU的所有计算单元SM，所以导致资源的闲置和利用率不高的问题，而MPS允许来自不同进程的kerne在同一个GPU上重叠运行，所以实现更好的资源利用率并缩短了任务执行的时间，大大提高了计算效率。

2）减少GPU上下文切换

类似于CPU上下文切换的概念，GPU也有上下文的概念，GPU上运行进程的切换，也伴随着进程在GPU上的上下文切换，引入MPS情况下，连接到同一个MPS server的client共享一组调度资源，从而消除了GPU在这些client之间调度时的GPU上下文切换开销。

3）减少GPU上下文存储空间

没有MPS的情况下，GPU上的每个CUDA应用进程都会在GPU上分配单独的存储和调度资源，而使用MPS之后，只需要为MPS server分配一份存储和调度资源，其他所有连接到MPS server的client共享这个一份资源，所以大大减少了进程在GPU上的上下文资源占用。

2.3 MPS架构

通常情况下，不同CUDA上下文进程或者任务只能通过GPU时间片调度器轮流执行，无法真正并行，当一个任务无法占满GPU的SM时，就会导致资源的闲置，而GPU整体的使用率很高但是真正计算核心的使用率却不高。

如下图，使用MPI在OS内启动了两个进程A（蓝色）和进程B（暗红色），每个进程都有自己的CUDA 上下文，虽然两个进程同时并发调度到一个GPU，但是每个MPI进程在整个GPU上被分配了一个串行调度的时间片，即实际两个进程没有真正的并行。

当使用MPS时，MPS server将会创建一个CUDA上下文，其他MPS client端的上下文将映射到server的Context，这样client切换不再需要GPU上 CUDA 上下文的切换，大大减少了上下文切换开销。各个客户端client提交的任务将由Server Process协调以确保同时调度，MPS client和MPS server之间的通信完全封装在CUDA API背后的CUDA驱动程序中，因此，MPS对MPI程序是透明的。

MPS控制守护进程负责MPS服务器的启动和关闭，控制守护进程最多只允许一个MPS服务器处于活动状态。

如下图，GPU开启MPS，系统启动了两个进程A（蓝色）和B（暗红色），MPS server进程会将两个进程的CUDA上下文映射到server的CUDA上下文中，两个进程并行执行，GPU上不存在上下文的切换，且从时间片调度上看，两个进程真正实现了并行执行。

NUMA

NUMA（Non-Uniform Memory Access，非一致性内存访问）

NUMA是多核/多CPU系统中的一种内存架构。 它将系统的内存划分为多个节点（NUMA Node），每

个节点绑定一部分 CPU 和本地内存。

特点是：CPU 访问本地内存速度快，而访问其他节点的远程内存会更慢。 因此，线程与数据的本地性 对性能非常关键。

1. NUMA 绑定 (numactl)：粗粒度的资源控制

numactl 是一个 Linux 命令行工具，用于在进程启动时强制指定它能使用的硬件资源范围。

  • 工作原理：
    • 绑定 CPU：你可以指定程序只能在某个 NUMA 节点的特定核心上运行。
    • 绑定内存：你可以指定程序只能在某个 NUMA 节点的本地内存中申请空间。
  • 策略示例：
    • --cpunodebind=0：强制程序在节点 0 的 CPU 上运行。
    • --membind=0：强制程序只从节点 0 分配内存。
    • --interleave=all：将内存交叉分配到所有节点上，适合那些访存极其巨大、单个节点内存带宽不足的程序。
  • HPC 意义：在你的集群环境中，如果一个 MPI 进程被分配到了节点 1，但它却跑去读节点 0 的内存，性能会因为跨节点通信而大幅下降。使用 numactl 可以从外部强制实现“本地化”。

2. 线程亲和性 (Thread Affinity)：防止核心“漂移”

线程亲和性（又称 CPU 绑定）是在线程级别进行的细粒度控制，通常由操作系统调度器或 OpenMP 库管理。

  • 工作原理：将特定的线程（Thread）锁定在特定的逻辑核心（Core）上，禁止操作系统为了“负载均衡”将线程从一个核心移动到另一个核心。
  • 为什么要绑定：
    • 保护缓存（Cache）：如果线程 A 在核心 1 运行，它的数据会填满核心 1 的 L1/L2 缓存。如果它被移动到核心 2，这些缓存就全失效了（Cache Miss），必须重新从内存读取。
    • 保持 NUMA 本地性：防止线程“漂移”到远离其数据所在内存节点的 CPU 上。
  • 实现方式：在 OpenMP 中，你可能常用环境变量 OMP_PROC_BIND=true 和 OMP_PLACES=cores 来实现这一点。

3. First-touch 策略：内存页的延迟分配机制

这是面试中最常考、也最容易在实际编程中掉坑的物理特性。

  • 工作原理：在 Linux 系统中，当你调用 malloc() 申请 10GB 内存时，系统并没有真正分配物理内存页。真正的物理内存是在 CPU 第一次访问（通常是写入）该内存地址时，才会被分配到该 CPU 所在的 NUMA 节点上。
  • 典型的“陷阱”：
    • 错误做法：在主线程（Rank 0）里用一个单线程循环初始化所有网格数据，然后开启多线程进行并行计算。结果：所有数据都会被挤在主线程所在的那个 NUMA 节点内存里，导致其他节点的线程在计算时全都在进行远程访问，造成严重的带宽瓶颈。
    • 优化做法：使用并行初始化。在计算开始前，先开多线程对数据进行初值化。这样，每个线程负责的那部分数据就会根据“First-touch”原则，自动分配到该线程运行的本地 NUMA 节点上。

2.这个循环在做什么？

这个循环实现的是 “网格跨步循环” (Grid-stride Loop) 的变体，专门用于数据搬运。

逻辑流程：

  1. **第一趟 (m = 0)**：
    • 线程根据自己的 threadIdx 找到初始行。
    • cache_idx = 0。
    • 执行 FETCH_FLOAT4，把 4 个数存进 load_a_cache[0, 1, 2, 3]。
  2. **第二趟 (m = a_tile_stride)**：
    • 线程向下跳跃 a_tile_stride 行。
    • cache_idx = 4 (1 * 4)。
    • 执行 FETCH_FLOAT4，把下 4 个数存进 load_a_cache[4, 5, 6, 7]。

在上文的代码中，我们在循环中使用了两次 __syncthreads() 来做线程同步，以防止不同线程之间的数据不一致。其中第一个 __syncthreads() 是为了保证写后读（Read-After-Write）的顺序性，这个是无法避免的。 关键在于后一个同步，它是为了防止部分线程还未读取 As 或者 Bs 中的内容，保证读后写（Write-After-Read）的顺序性。它本质上是因为我们在不同迭代中使用了同一块空间来保存我们所需的数据，这两次迭代中的数据之间并不存在真正的依赖关系。如果我们将其写入到其他地址上，那么就不需要使用同步了。 这种方式称为数据预取，又可以称为双缓存（Double Buffering）。我们可以申请两倍所需的空间，用于在迭代中交替使用，从而省去后一次线程同步的操作。
